# План переноса Everyday_test в Jenkins

Обновлено: 2026-05-06

## 1) Цель

Перенести выполнение `Everyday_test` из GitHub-only рантайма в Jenkins без потери:

- стабильности pass-rate,
- покрытия по провайдерам,
- покрытия desktop + mobile,
- корректной работы алертов,
- качества Allure-отчетности.

Ключевое условие: сохранить паритет команд запуска с текущими pytest-прогонами.

## 2) Область работ

Входит в объем:

- провайдерные сценарии `big_landing_code.py`,
- режимы `core`, `variants`,
- desktop и mobile execution profile,
- smoke и full наборы провайдеров,
- общий кеш Playwright браузеров в Jenkins.

Не входит в первую фазу:

- глубокая кастомизация Jenkins-плагинов,
- немедленная полная замена текущих GitHub-оркестраторов.

## 3) Модель запуска

Jenkins должен выполнять ту же логику, что и текущий CI:

- `python -m pytest big_landing_code.py ... --provider=<provider>`
- desktop: `--browser=chromium|firefox|webkit`
- mobile:
  - `--browser=chromium --execution-profile=mobile-chromium`
  - `--browser=webkit --execution-profile=mobile-webkit`

## 4) Политика кеша Playwright

Используется постоянный путь:

- `PLAYWRIGHT_BROWSERS_PATH=/var/lib/jenkins/cache/ms-playwright`

Правила:

- создавать каталог кеша при отсутствии,
- устанавливать браузер только если его нет в кеше,
- не выполнять установку браузеров при каждом билде.

Ожидаемый эффект:

- меньшее стартовое время,
- меньше расхода диска,
- более предсказуемая нагрузка по памяти на Jenkins-нодах.

## 5) Параметры Jenkins job (зафиксированный минимум)

- `PROVIDER_SCOPE`: `all | smoke | mts | beeline | megafon | t2 | rostelecom | domru`
- `SITE`: опциональный фильтр по сайту
- `SERVICE_MODE`: `core | variants | all`
- `RUN_CHROMIUM`: bool
- `RUN_FIREFOX`: bool
- `RUN_WEBKIT`: bool
- `RUN_MOBILE_CHROMIUM`: bool
- `RUN_MOBILE_WEBKIT`: bool
- `BLOCKING_PROFILE`: `none | adblock-mvp`
- флаги алертов (`ALERT_ERRORS`, `ALERT_AGGREGATES`, `ALERT_SUMMARY`, `ALERT_RECOVERED`)

Guardrail:

- минимум один браузерный/профильный флаг должен быть включен.

## 6) Фазы внедрения

### Фаза A: dry-run на быстрых нестабильных кейсах

Цель:

- `domru` + `t2`,
- mobile `core`,
- `chromium` и `webkit`.

Критерии успеха:

- запускаются все ожидаемые тесты,
- нет постоянной перезакачки браузеров,
- формируются артефакты Allure.

### Фаза B: полный mobile-проход по провайдерам

Цель:

- все провайдеры,
- `core + variants`,
- mobile chromium + webkit.

Критерии успеха:

- паритет с последними зелеными mobile orchestrator прогонами,
- стабильный выбор города и popup-флоу в мобильных профилях.

### Фаза C: полный прогон desktop + mobile

Цель:

- все провайдеры,
- согласованный набор desktop браузеров + mobile профилей,
- полный цикл алертов и отчетности.

Критерии успеха:

- pass-rate и время прогона в согласованном коридоре,
- отсутствие архитектурных регрессий между desktop и mobile.

### Фаза D: операционное переключение

Цель:

- Jenkins становится основным планировщиком nightly/full запусков.

Критерии успеха:

- повторяемые зеленые прогоны,
- у команды есть наблюдаемость и прозрачный мониторинг.

### Фаза E: очистка

Цель:

- удалить временные пилотные артефакты после подтвержденной стабильности.

Критерии успеха:

- в рантайме нет зависимостей от временных файлов,
- контекстные документы обновлены.

## 7) Definition of Done

Миграция считается завершенной, когда:

- Jenkins запускает smoke и full срезы через параметры,
- кеш браузеров стабильно используется,
- Allure-артефакты формируются на каждом запуске,
- mobile и desktop используют единую архитектуру provider-конфигов,
- алерты работают как ожидается,
- временные файлы убраны после валидации.

## 9) Важно для продукта и команды

- Миграция не меняет бизнес-логику провайдерных тестов.
- Mobile-тесты используют те же provider-конфиги, что и desktop-тесты.
- Архитектура централизована: изменения конфигов в одном месте автоматически подхватываются всеми execution profile.
