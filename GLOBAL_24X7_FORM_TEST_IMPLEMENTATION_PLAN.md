## 2. Цель первой итерации
Собрать рабочий непрерывный автотест форм по всем брендам, который:
- берёт URL из внешних файлов `urls/<brand>.txt`,
- ходит напрямую по URL (без навигации по сайту),
- проверяет формы и submit/redirect (`/thanks` или `/tilda/form1/submitted`),
- выполняет повторный заход со сменой города (кроме `business`),
- отправляет алерты и сводки по новым правилам,
- сначала проходит пилот в GitHub Actions, затем переносится в Jenkins.

## 2.1 Порядок CI-внедрения (уточнение)
- Этап проверки №1: GitHub Actions (тестовый CI-контур для валидации логики).
- Этап проверки №2: Jenkins (после подтверждения логики и стабильности в GitHub Actions).

## 2.2 Обязательная логика выполнения (согласовано)
1. Тест заходит на URL напрямую из `urls/<brand>.txt`.
2. Для каждого URL тест проверяет наличие утвержденных форм из внешнего конфига ожиданий форм.
3. Для каждого типа формы на странице выполняется одна отправка заявки:
   - не дублируем отправку по всем одинаковым кнопкам,
   - берем один представитель на тип формы (для ускорения и предсказуемости диагностики).
4. Один запуск выполняется для одного бренда:
   - пройти все URL выбранного бренда,
   - затем переходить к следующему бренду отдельным шагом/джобой.
   Это нужно для оптимизации и быстрой локализации падений.

## 3. Порядок работ (этапы)
### Этап 0. Baseline нового репозитория
- Зафиксировать baseline как стартовую точку в `Big_landing_test`.
- Убедиться, что в стартовом коммите нет временных артефактов прогона (кеши, `allure-results`, `tmp`, `__pycache__`).
- Проверить базовую collect-совместимость (`pytest --collect-only`) до рефакторинга.

### Этап 1. Новый источник URL
- Добавить директорию `urls/`:
  - `mts.txt`, `beeline.txt`, `megafon.txt`, `t2.txt`, `rostelecom.txt`, `domru.txt`, `ttk.txt`.
- Реализовать загрузчик URL:
  - один URL на строку,
  - строки с `#` игнорируются,
  - пустые строки игнорируются,
  - опциональная дедупликация внутри файла.
- Подготовить первичное наполнение файлов по согласованному списку.

### Этап 2. Рефакторинг раннера под прямой обход страниц
- Перевести запуск на сценарий "прямой заход по URL из файла".
- Добавить паузу не менее 3 секунд между страницами.
- Правила недоступности страницы:
  - `HTTP >= 400` -> skip страницы с ошибкой в отчёте,
  - timeout навигации: повтор через 10 сек,
  - если после повтора суммарно > 30 сек -> skip страницы с ошибкой в отчёте.

### Этап 3. Формы и шаги проверки
- Сохранить текущую стабильную бизнес-логику заполнения/submit, расширив её под новый набор форм.
- Покрыть типы форм:
  - `profit`, `connection`, `checkaddress`, `undecided`, `business`, `moving`, `express-connection`.
- Для `moving` добавить поддержку с нуля.
- Для форм с `Place` оставить отправку по каждому варианту.
- Для `business`:
  - без сайджеста,
  - без шага смены города.
- Ввести слой ожиданий форм по URL (внешний конфиг):
  - определяет, какие типы форм обязательны на конкретной странице,
  - участвует в проверке "форма должна быть/не должна быть" до submit-сценария.

### Этап 4. Повторный заход и смена города
- После первого успешного submit по форме делать повторный прямой заход на тот же URL.
- Выполнять смену города в форме (кроме `business`).
- Проверять, что после смены города сайджест улицы соответствует новому городу по согласованной эвристике.

### Этап 5. Оркестрация 24/7 по брендам
- Реализовать непрерывную цепочку брендов:
  - `MTS -> Beeline -> Megafon -> T2 -> Rostelecom -> Domru -> TTK -> повтор`.
- Ввести флаг `enable_continuous_loop` для включения/выключения цепочки.
- Поддержать ручной запуск:
  - одного бренда,
  - полного цикла.

### Этап 6. Алерты и сводки
- Поддержать категории переключателей:
  - `alert_errors`,
  - `alert_aggregates`,
  - `alert_summary`,
  - `alert_daily_summary`,
  - `alert_recovered`.
- Логика уведомлений:
  - единичные (1-5 страниц),
  - агрегированные (>5 страниц по лендингу),
  - массовые (системная ошибка на нескольких брендах/лендингах),
  - recovered-уведомления после восстановления.
- Повторы по "живущей" ошибке:
  - 1-й алерт сразу,
  - затем примерно через 30 минут,
  - затем примерно через 1 час после предыдущего,
  - далее раз в сутки до восстановления.
- Дневная сводка:
  - 09:00 и 17:00 МСК.

### Этап 7. GitHub Actions pilot (обязательный перед Jenkins)
- Собрать тестовый workflow-контур в GitHub Actions для проверки:
  - новой логики обхода URL из `urls/*.txt`,
  - поведения теста по формам,
  - новой схемы алертов/сводок.
- Запуски в этом этапе считаются валидационными, логика уточняется и согласуется по результатам.
- До переноса в Jenkins фиксируем согласованную "рабочую" версию логики.

### Этап 8. Jenkins
- Перенести в Jenkins уже согласованный и проверенный в GitHub Actions контур.
- Закрепить Jenkins как основной рантайм.
- Добавить параметры для:
  - бренда/скоупа,
  - непрерывного режима,
  - профиля браузера/мобильного профиля (если применимо),
  - категорий алертов.
- Подготовить минимальный dry-run и smoke-run гейт.

### Этап 9. Документация
- Обновить `README.md` и/или отдельный runbook:
  - что проверяется,
  - как править `urls/*.txt` без кода,
