# План реализации: Global 24/7 Form Test (Big_landing_test)
Обновлено: 2026-05-12

## 1. Зафиксированные рамки (согласовано)
- Базу берём из `Everyday_test`, дорабатываем через рефакторинг.
- ClickHouse: не реализуем в текущей итерации.
- Проверка заявок в БД: не реализуем в текущей итерации.
- Проверка соответствия региона URL/формы: заморожена в 1-й итерации.
- Сначала валидируем логику в GitHub Actions, потом перенос в Jenkins.

## 2. Цель текущей итерации
Собрать стабильный контур глобального теста форм, который:
- читает URL из `urls/<brand>.txt`;
- запускается по `provider x form_suite` в GitHub Actions;
- корректно отделяет боевые падения от инфраструктурных;
- готов к следующему шагу: оркестрация 24/7.

## 3. Текущий статус (факт на 2026-05-12)

### 3.1 Уже реализовано
- `R0`: baseline репозитория зафиксирован.
- `R1`: добавлены `urls/*.txt` и загрузчик `config/url_loader.py`.
- `R2`: реализован запуск по URL-бренду (`--url-brand`), прямой обход URL.
- `R3`: расширена форма-логика, добавлены новые типы/allowlist-ожидания по form suite.
- В `global-forms-pilot.yml` внедрён режим `provider x form_suite`.
- Добавлены shard-параметры `url_shard_total/url_shard_index`.
- Добавлены группы провайдеров:
  - `big_two`: `mts + beeline`
  - `rtk_megafon`: `rostelecom + megafon`
  - `small_pool`: `t2 + domru + ttk`
- Добавлен опциональный режим `provider_sequence` (последовательный запуск брендов), основной режим остаётся матричным.
- В `Build Matrix` добавлена предфильтрация пустых комбинаций `brand x suite` (чтобы не падать на collection error).
- В navigation добавлена классификация SSL/сетевых `ERR_*` как site unavailable (critical skip), чтобы убрать ложный шум.

### 3.2 Точечные стабилизации, внесённые по ходу
- Soft retry по дому для проблемных URL (`1 -> 2 -> 7`) с безопасным fallback.
- Усилена очистка поля дома между попытками (жёсткий reset ввода).
- Добавлен селектор `.autocomplete-item` в список сайджеста.

### 3.3 Подтверждённые наблюдения
- Пустые комбинации allowlist действительно встречаются на части брендов/форм и теперь корректно отсекаются на этапе построения матрицы.
- Часть падений в прогонах являются фактическими боевыми проблемами сайтов (например `popup_not_found`), а не дефектами CI-логики.

## 4. Что изменилось в подходе относительно исходного плана
- Ранее предполагался акцент на последовательную цепочку брендов; по факту для скорости и диагностики выбран основной режим: матричный `provider x form_suite`.
- Для крупных наборов URL используется шардинг вместо полного последовательного прогона.
- Массовые/сложные алерты пока не форсируем, пока не запущен стабильный 24/7-контур.

## 5. Актуальный порядок дальнейших работ

### Этап E. Jenkins и финализация

## 7. Definition of Done для ближайшего инкремента
- Прогоны `small_pool` и `rtk_megafon` не падают на технических пустых комбинациях suite.
- Инфраструктурные SSL/сеть инциденты не валят весь прогон как продуктовые дефекты.
- По каждому падению можно однозначно определить: боевой инцидент сайта или техничный skip.
- План и документация синхронизированы с реальным поведением CI.
