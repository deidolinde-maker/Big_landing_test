# План реализации: Patch Branch 4 (Mobile Profile Rollout)
Дата: 2026-04-29

## Цель ветки
Внедрить мобильный профиль тестирования как отдельный и безопасный контур, не влияющий на стабильность desktop-прогонов.

Ключевой порядок rollout:
1. Сначала `mobile-webkit` пилот на ограниченном скоупе.
2. Затем оптимизация по итогам пилота.
3. После подтверждения стабильности — раскатка на остальных провайдеров.

## Критерии mobile-профиля (Definition of Success)
1. Mobile-логика независима от desktop и не снижает стабильность основного скрипта.
2. Mobile существует как отдельный workflow-профиль и работает по основному сценарию.
3. Мобильные особенности учитываются только в mobile execution profile.
4. Telegram-алерты подключены полностью, как в основном тесте.
5. Для mobile есть отдельные Allure-отчёты и логи.
6. Есть ручной запуск как полного прогона, так и точечного (`provider`/`site`) по аналогии с core/place/adblock.

## KPI перехода между этапами
1. Переход от `mobile-webkit` пилота к раскатке на остальных провайдеров:
  - не менее 3 последовательных green run на пилотном скоупе;
  - отсутствие desktop регрессий на контрольном smoke.
2. Переход к расширению матрицы сценариев (`variants`, `adblock`):
  - стабильный green full core по целевому провайдеру в mobile-профиле.

## Архитектурный принцип
Используем один основной тестовый скрипт с execution profiles:
- `desktop` (по умолчанию);
- `mobile-chromium`;
- `mobile-webkit` (добавляется в этой ветке).

Мобильные обходы и локаторы включаются только при mobile-профиле.

## Ограничения безопасности
1. Никаких автозапусков mobile до завершения пилота и стабилизации.
2. Никаких изменений desktop workflow в рамках mobile rollout (кроме совместимых чтений общих функций).
3. Любая mobile-правка должна проходить desktop smoke-check до merge.

## Контроль длительности прогонов (SLA-lite)
1. На текущем этапе mobile прогоны остаются ручными и отдельными, поэтому не блокируют desktop CI.
2. Длительность mobile-run фиксируется как метрика наблюдения (без fail-gate).
3. При аномальном росте длительности (например, +30-40% к базовому времени) создаётся tech-alert на оптимизацию.

## Пошаговый план реализации

### P4-R0. Freeze baseline и контрольный эталон
- Объём:
  - зафиксировать baseline desktop стабильности (текущие рабочие provider workflow);
  - зафиксировать baseline mobile-chromium пилота (MTS).
- Критерий приёмки:
  - есть подтверждённые успешные baseline прогоны для сравнения до/после.
- Проверка:
  - desktop smoke на 1-2 MTS сайта;
  - mobile-chromium smoke на `mts-home.online`.

### P4-R1. Введение execution profile `mobile-webkit`
- Объём:
  - расширить `conftest.py` новым профилем `mobile-webkit`;
  - задать device preset для WebKit мобильной эмуляции;
  - добавить строгую валидацию комбинаций `--execution-profile` и `--browser`.
- Критерий приёмки:
  - профиль запускается только с поддерживаемым браузером;
  - при неверной комбинации получаем читаемую ошибку на старте.
- Проверка:
  - `pytest --collect-only` + негативные проверки неправильных комбинаций.

### P4-R2. Изоляция mobile-логики и mobile-overrides
- Объём:
  - вынести mobile-специфику (бургер, оверлеи, tap/focus fallback) в отдельные функции/конфиг;
  - привязать активацию только к `mobile-*` профилям.
- Критерий приёмки:
  - desktop-код не содержит активной mobile-ветки исполнения;
  - mobile-локаторы не применяются в desktop.
- Проверка:
  - desktop smoke остаётся зелёным;
  - mobile pilot использует мобильные обходы в логах.

### P4-R3. Telegram parity для mobile
- Объём:
  - убедиться, что mobile использует ту же схему `tech/step/final` алертов;
  - проверить payload и полезность диагностики на mobile ошибках.
- Критерий приёмки:
  - алерты mobile эквивалентны основному тесту по полноте.
- Проверка:
  - тестовый прогон с контролируемой ошибкой;
  - валидация текста и ссылок в уведомлениях.

### P4-R4. Отдельная mobile-отчётность (Allure + artifacts + pages)
- Объём:
  - закрепить отдельные директории `allure-results-*mobile*`;
  - отдельная публикация в `gh-pages` по mobile путям;
  - артефакты результатов mobile отдельно от desktop.
- Критерий приёмки:
  - mobile-отчёты не перетирают desktop-отчёты;
  - результаты доступны по стабильному URL mobile-раздела.
- Проверка:
  - ручной прогон mobile workflow с публикацией отчёта.

### P4-R5. Пилот `mobile-webkit` (ограниченный скоуп)
- Объём:
  - создать ручной workflow пилота `provider-mts-mobile-webkit` (или эквивалентный шаг в mobile workflow);
  - запуск только на MTS и только в ручном режиме;
  - поддержать фильтры `site`, `service_mode`, `blocking_profile`.
- Критерий приёмки:
  - пилот проходит основной сценарий на целевых сайтах без влияния на desktop.
- Проверка:
  - точечный прогон `--site`;
  - провайдерный core прогон по MTS.

### P4-R6. Оптимизация после `mobile-webkit` пилота
- Объём:
  - собрать и разобрать все падения пилота;
  - доработать только mobile-overrides (без изменения desktop-поведения);
  - зафиксировать known-issues и допустимые исключения по сайтам (временный allowlist).
- Критерий приёмки:
  - повторные прогоны показывают улучшение стабильности;
  - нет desktop регрессий.
- Проверка:
  - повторные mobile-webkit прогоны на том же скоупе;
  - desktop smoke после каждой существенной правки.

### P4-R7. Масштабирование mobile на остальных провайдеров
- Объём:
  - добавить provider-specific mobile workflow по шаблону MTS;
  - сохранить ручной режим запуска;
  - обеспечить поддержку точечного запуска `provider/site`.
- Критерий приёмки:
  - любой провайдер можно запустить отдельно в mobile профиле;
  - логика запуска аналогична текущим provider workflow.
- Проверка:
  - по одному smoke-прогону на каждого провайдера в mobile профиле.

### P4-R8. Расширение матрицы сценариев
- Объём:
  - поддержать для mobile те же сценарные режимы: `core`, `variants(place)`, `blocking_profile` (например `adblock-mvp`);
  - запускать режимы отдельными ручными прогонками.
- Критерий приёмки:
  - mobile покрывает тот же сценарный контракт, что и desktop-пайплайн.
- Проверка:
  - отдельные успешные прогоны `core` и `variants`;
  - отдельный прогон с `blocking_profile=adblock-mvp`.

### P4-R9. Release guardrails и rollback-механика
- Объём:
  - добавить feature-gates на mobile шаги/профили;
  - задокументировать быстрый rollback mobile workflow без отката desktop.
- Критерий приёмки:
  - при любой нестабильности mobile можно отключить за один безопасный commit.
- Проверка:
  - тестовый dry-run rollback на ветке.

### P4-R10. Финализация документации
- Объём:
  - обновить `README.md`:
    - как запускать mobile вручную;
    - как запускать full provider и точечный site;
    - как читать mobile Allure и mobile логи;
    - как интерпретировать mobile Telegram alerts.
- Критерий приёмки:
  - новый участник команды запускает mobile прогон без чтения исходников.
- Проверка:
  - smoke по инструкции "с нуля".

## Контрольные гейты на каждом шаге
1. `pytest --collect-only` проходит без ошибок.
2. Desktop smoke зелёный после каждой mobile-правки.
3. Mobile целевой smoke зелёный на затронутом скоупе.
4. Allure mobile публикуется отдельно.
5. Telegram alerts отправляются в полном формате.

## Риски и меры
1. Риск: mobile-обход сломает desktop поведение.
Мера: строгая активация mobile-кода только через `execution_profile`.

2. Риск: рост флейков из-за адаптивной верстки и бургер-меню.
Мера: provider-specific mobile-overrides + явные логи pre-steps.

3. Риск: смешивание desktop/mobile отчётов и артефактов.
Мера: отдельные директории результатов и отдельные `gh-pages` пути.

4. Риск: усложнение поддержки workflow.
Мера: единый шаблон provider-mobile workflow + общие env/аргументы.

## Формат отчётности по выполнению
Для каждого шага `P4-Rx` фиксируется:
- commit hash;
- что сделано;
- что проверено;
- результат (pass/fail);
- остаточный риск;
- следующий шаг.
